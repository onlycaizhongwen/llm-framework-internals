开场

今天分享的是 NousResearch/hermes-agent。先定个位：它不是 LangGraph、Dify、Haystack 那种“给开发者搭 Agent/RAG/Workflow 的框架”，而是一个完整的个人 AI Agent 产品工程。

它关心的是：一个 Agent 如何同时跑在 CLI、Gateway、ACP、Cron、Desktop 里；长期会话如何恢复和检索；工具执行如何审批；Gateway 重启后任务如何继续；插件、技能、MCP 如何接入但不污染核心循环。

核心观点

Hermes 的源码价值，不在于它抽象出一个漂亮的 Agent 框架，而在于它展示了一个长期在线、跨入口、可扩展、可恢复、要控制成本和风险的个人 Agent 产品，工程上到底需要补齐哪些层。

  1. 多入口接入层：CLI、Gateway、ACP、Cron、Desktop。
  2. Agent Core：AIAgent + conversation_loop。
  3. 能力边缘：Tool Registry、Toolsets、Plugin、Skill、MCP。
  4. 长期状态：SessionDB、FTS/trigram、memory、compression lineage。
  5. 产品投递与恢复：DeliveryRouter、pending queue、resume_pending。
两个关键词：Prompt cache first 和 Core narrow waist。

三条主线

主线一：多入口最终回到同一个 Agent Core

CLI、TUI、Messaging Gateway、ACP、Cron、Desktop 看起来分散，但最终都尽量回到同一个核心：AIAgent + conversation_loop。入口处理协议差异，核心处理 Agent 生命周期。

推荐话术：Hermes 的入口很多，但核心只有一个。入口负责把不同场景接进来，Agent Core 负责把一轮任务跑完整。

主线二：工具能力放在边缘，但安全边界必须前置

工具通过 Tool Registry 管理，schema、handler、toolset 有统一来源。插件和 MCP 可以扩展工具，但仍然进入统一 registry。危险命令审批、hardline blocklist、session_key 隔离、secret redaction 让工具执行保持可控。

推荐话术：Hermes 的工具系统不是单纯为了让模型多做事，而是为了让模型在可控边界内做事。

主线三：长期状态和 Gateway 恢复是产品复杂度的核心

SessionDB 支撑会话、消息、工具调用、全文检索、中文检索和压缩 lineage。Gateway 通过 running agents、pending queue、resume_pending、generation check 处理连续消息、长任务和重启恢复。

推荐话术：Hermes 和普通 Agent demo 的差异，不在模型调用，而在长期运行时的会话一致性、恢复、排队、审批和投递。

真实案例

案例一：CLI 里让 Hermes 修复仓库 bug

hermes -w -q "检查当前仓库测试失败原因并修复"

CLI 解析参数，创建隔离 worktree，创建或恢复 AIAgent，进入 conversation_loop。模型通过工具 schema 调用 file/search/terminal，工具执行前经过 hook、edit approval、dangerous command approval，最终结果持久化并触发后台 review。

案例二：Telegram 里创建定时总结任务

Telegram adapter 标准化消息，GatewayRunner 生成 session_key，Agent 创建 cron job。scheduler 每 60 秒 tick，执行时禁用不适合非交互场景的工具，结果通过 DeliveryRouter 回到 Telegram。

案例三：Gateway 长任务执行中重启

Gateway 在 shutdown/restart 前标记 resume_pending，drain 超时后中断剩余 Agent。重启后 _schedule_resume_pending_sessions() 合成空事件，让原会话继续执行，并在当前轮结束后处理 pending queue。

和 LangGraph / CrewAI / AutoGen 的区别

项目更适合讲什么
LangGraph状态图、节点、边、checkpoint、中断恢复。
CrewAIRole Agent、Task、Crew、Process，多角色协作。
AutoGen多 Agent 对话、消息路由、工具调用 runtime。
Hermes Agent长期个人 Agent 产品，多入口、工具安全、状态恢复、平台投递。
推荐话术：LangGraph 讲怎么编排 Agent，CrewAI 讲怎么分配角色和任务，AutoGen 讲怎么让多个 Agent 对话协作，而 Hermes 讲的是一个 Agent 真正长期在线以后，产品工程要补哪些层。

总结

Hermes 给我们的启发有三点：

  1. Agent 产品的复杂度不只在模型调用，而在多入口、多会话、长时间运行、工具安全、状态恢复和平台投递。
  2. 核心要保持窄腰，入口、provider、tool、plugin、skill、MCP、platform adapter 都放在边缘扩展。
  3. 长期在线 Agent 必须默认考虑风险，审批、硬阻断、会话隔离、脱敏、恢复都属于产品可用性的基础。
推荐结尾：Hermes 的源码价值，不在于它给了我们一个新的 Agent 框架 API，而在于它把一个长期运行、跨入口、可恢复、可扩展、要控制成本和风险的个人 Agent 产品，拆成了可以学习的工程层次。
