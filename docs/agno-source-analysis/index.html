Agno 源码架构精读

Agno 的定位不是“再封装一个 agent.run()”，而是一个 Agent platform SDK：用代码定义 Agent、Team、Workflow，再通过 AgentOS 把这些组件暴露为 REST、WebSocket、MCP、界面接口、调度、权限和可观测能力。

1. 总体结论

Agno 最值得讲的主线是：把 Agent 从“本地脚本对象”提升成“可运行、可管理、可继续、可审批、可对外提供 API 的平台组件”。

一句话分享：Agno 是代码优先的 Agent 平台 SDK。它既有单 Agent loop，也有多 Agent Team、确定性 Workflow，并用 AgentOS 把这些能力服务化。
层次核心对象解决的问题
单 Agent 能力层Agent模型调用、工具、知识、记忆、Hook、结构化输出、session。
多 Agent 协作层Teamleader 调度多个 Agent 或嵌套 Team，支持 coordinate、route、broadcast、tasks。
确定性流程层Workflow / Step用步骤、条件、循环、并行、路由组织长流程，并支持 HITL 暂停恢复。
平台服务层AgentOSFastAPI、WebSocket、MCP、接口、权限、调度、数据库生命周期。

6. Workflow 流程

flowchart TB
    Start["workflow.run / arun"] --> Init["初始化 workflow<br/>session、state、metadata、run_context"]
    Init --> Prepare["准备 steps<br/>函数、Agent、Team、嵌套 Workflow 都包装成 Step"]
    Prepare --> StepKinds["Step 类型<br/>Step / Steps / Parallel / Loop / Condition / Router"]
    StepKinds --> Execute["按流程执行<br/>传递 StepInput、StepOutput、媒体和 state"]
    Execute --> HITL{"步骤是否需要人工介入？"}
    HITL -->|"确认或输入"| Pause["生成 StepRequirement<br/>暂停并保存运行状态"]
    Pause --> Continue["continue_run<br/>带用户输入、审批或重路由选择恢复"]
    Continue --> Execute
    HITL -->|"无需暂停"| Review{"需要输出审核或错误处理？"}
    Review -->|"需要"| Pause
    Review -->|"通过"| Next["进入下一个步骤"]
    Next --> Done{"步骤完成？"}
    Done -->|"否"| Execute
    Done -->|"是"| Store["保存 WorkflowSession、metrics、run output"]
    Store --> End["返回 WorkflowRunOutput 或事件流"]

Agno 的 Workflow 更像“业务流程编排器”，不是 LangGraph 那种以 checkpoint-first 状态图为核心的通用图运行时。它的强项是把函数、Agent、Team 和嵌套 Workflow 放进同一套 Step 体系。

9. 真实示例

企业财务审批 Agent 平台：员工提交付款审核，Agent 查合同、发票、付款历史和风险规则；高风险付款工具用 @approval(type="required") 阻塞审批；AgentOS 暴露 API 与审批入口。

9.1 最小代码示例：Agent + tool + approval + AgentOS

下面这段不是为了展示完整业务代码，而是说明 Agno 的核心组合方式：先用代码定义 Agent 和工具，再用 @approval 给高风险工具加审批边界，最后用 AgentOS 把它服务化。

from agno.agent import Agent
from agno.approval import approval
from agno.models.openai import OpenAIChat
from agno.os import AgentOS
from agno.tools import tool


def lookup_invoice(invoice_id: str) -> str:
    return f"invoice={invoice_id}, amount=128000, vendor=ACME"


@approval(type="required")
@tool()
def release_payment(invoice_id: str) -> str:
    return f"payment released for {invoice_id}"


finance_agent = Agent(
    id="finance-approval",
    name="财务付款审批 Agent",
    model=OpenAIChat(id="gpt-4.1"),
    tools=[lookup_invoice, release_payment],
    instructions=[
        "先核对发票、供应商和金额风险。",
        "只有确认符合规则时才调用 release_payment。",
    ],
    markdown=True,
)

agent_os = AgentOS(
    id="finance-agent-os",
    agents=[finance_agent],
    enable_mcp_server=True,
    authorization=True,
)

app = agent_os.get_app()
分享时可以强调：这段代码背后不是简单函数调用。release_payment 会被标记成需要审批的工具；AgentOS 会把 Agent 暴露成服务端资源；如果 run 因审批暂停，外部系统可以通过 REST / UI / MCP 处理后再 continue_run。
知识库问答 + 后台学习：售后工程师询问设备故障，Agent 通过 knowledge 检索手册和维修记录，同时后台 learning / culture 抽取可复用经验。
多角色研究团队：市场研究员、技术分析员、财务分析员分别处理，Team(mode=coordinate) 由 leader 汇总；如果只选一个专家用 route，多视角并行用 broadcast。
订单处理 Workflow：识别诉求、查订单、判断退款规则、人工确认、写回工单。普通步骤用函数，推理步骤用 Agent，复杂分支用 Router 或 Condition。

LangGraph更像 checkpoint-first 状态图运行时；Agno 更像 Agent/Team/Workflow 平台 SDK。复杂状态内核用 LangGraph，外层 API、审批、界面、MCP 可参考 Agno。

PydanticAI更强调 typed deps、tool schema、结构化输出和 evals；Agno 更强调运行平台和状态治理。PydanticAI 做 typed inner agent，Agno 或 LangGraph 做外层服务/流程。

11. 核心设计思想

  1. 平台化优先：AgentOS 把 Agent 作为可被 API、UI、MCP 和调度系统管理的资源。
  2. 分层抽象：Agent 处理单体推理，Team 处理多 Agent 协作，Workflow 处理确定性步骤，AgentOS 处理服务化和治理。
  3. 状态一等公民：session、run、session_state、metrics、checkpoint、continue_run 贯穿 Agent、Team、Workflow。
  4. HITL 内建：审批、确认、用户输入、输出审核是 tool 和 step 的执行语义。
  5. 生产治理内建：auth、RBAC、tracing、scheduler、interfaces、MCP 都进入 AgentOS。
局限性：如果只想要最精细 checkpoint 状态图，LangGraph 更专注；只想要 typed business agent，PydanticAI 更轻；重 RAG ingestion 和检索工程，Haystack / LlamaIndex 更专业；低代码产品化入口，Dify 更合适。

12. 分享口径

开场可以这样讲：Agno 这类框架的价值，不在于它能不能调用一个模型，而在于它把 Agent 运行时升级成平台组件。它有单 Agent loop，有多 Agent Team，有确定性 Workflow，还有 AgentOS 把这些组件变成 API、WebSocket、MCP、审批、调度和权限体系。

  1. Agent 主线：模型、工具、记忆、知识、hook 和结构化输出如何组成一次 run。
  2. Team / Workflow 主线：多 Agent 协作和确定性业务流程如何复用 session、HITL、持久化能力。
  3. AgentOS 主线：为什么生产环境需要 API、权限、调度、MCP、接口和数据库生命周期，而不只是 agent.run()。
