Graphiti 源码架构精读

分析对象：sources/graphiti，固定源码提交 62ff03ac5662d288ebd9f6aafb70d6ae4070c632。参考 mem0 的分析方式，但 Graphiti 的核心是 temporal context graph：episode、entity、fact edge、时间失效和混合检索。

总体结论

Graphiti 是 Zep 背后的开源 temporal context graph engine。它不是只保存一组 memory facts，而是把输入保存成 EpisodicNode，再抽取实体和关系；每条关系边保存 episodes、valid_at、invalid_at、expired_at，所以能表达“事实什么时候成立、什么时候被新事实替代、证据来自哪里”。

分享口径：mem0 更像轻量 memory layer；Graphiti 更像带时间和溯源能力的图谱记忆内核；Zep 则是托管化 context graph 平台。

Mermaid 源文件：ingestion-flow.mmd

Graphiti.add_episode() 是写入主流程。它接收 episode_body、reference_time、entity_types、edge_types、saga 等参数，然后完成抽取、去重、失效和保存。

证据：graphiti.py:980 是入口；:1099-1138 取 previous episodes、创建 episode、抽取/resolve nodes；:1138-1167 抽取/resolve edges 和 invalidated edges；:1168-1221 保存并记录 span。

步骤细节为什么需要
取 previous episodes没有传 previous_episode_uuids 时，按 reference_time 拉近期 episode。新输入经常省略主语或上下文，抽取实体/事实需要最近对话补全语义。
group_id 分区为空时用 driver 默认分区；传入时校验并 clone driver 到对应 database。多用户、多租户或多业务图要隔离，避免事实串图。
entity_types / edge_types业务可以传 Pydantic model 和 edge type map。不把 schema 完全交给 LLM 自由发挥，降低抽取漂移。
saga / NEXT_EPISODEepisode 可关联到 saga，并按时间串起来。对连续任务、会话、故事线做时序追踪，方便按链路理解上下文。
tracing span记录 episode、node、edge、invalidated_count、duration。这条链路重且多 LLM 调用，生产上必须可观测。

主流程二：Temporal Fact Invalidation

Graphiti 最核心的动作不是“覆盖旧记忆”，而是让旧 fact 失效。这样它既能回答现在，也能保留历史。

elif edge_valid_at_utc < resolved_edge_valid_at_utc:
    edge.invalid_at = resolved_edge.valid_at
    edge.expired_at = edge.expired_at if edge.expired_at is not None else utc_now()
    invalidated_edges.append(edge)
为什么这么设计：用户偏好、企业规则、项目上下文都会变化。旧事实如果直接删除，就失去历史；如果不处理，又会污染当前答案。Graphiti 用 invalid_at 表达“旧事实到此为止”。

证据：edge_operations.py:545-573 写入 invalid_at/expired_at；edge_operations.py:623-848 用 LLM 判重和识别 contradicted facts。

类型去重/失效路径设计原因
EntityNode语义候选、确定性相似、LLM resolve。实体名容易同义、别名、大小写变化，先缩小候选再让 LLM 判断更稳。
EntityEdgeexact fact 快速去重，同端点 related edges，全图 invalidation candidates。fact 既可能重复，也可能语义矛盾；只查同端点会漏冲突，只查全图又太贵。
Attributes只在匹配自定义 schema 时抽结构化属性。避免 LLM 生成无约束属性污染实体/关系。
Embeddings新节点和新边在 resolve 前生成 embedding。候选召回、相似度排序和搜索都依赖向量。

基础入口 Graphiti.search() 返回 facts；高级入口 Graphiti.search_() 返回 SearchResults，可同时包含 edges、nodes、episodes、communities。

证据：graphiti.py:1527 基础 search；:1584 高级 search_；search.py:98-244 并行 scope search；search.py:262-462 edge search 的 BM25/cosine/BFS/rerank；search_config_recipes.py:33-109 combined hybrid recipes。

机制适合解决什么为什么不是只用向量
BM25 / fulltext代码名、产品名、人名、精确术语。embedding 对专有名词和短查询不一定稳定。
Cosine similarity语义相近但字面不同的问题。用户查询和 fact 表述常常不完全一致。
BFS已有中心节点或候选节点附近的关系扩展。图谱记忆的价值在关系，不能只看文本相似度。
RRF多路召回的轻量融合。不需要训练，能把不同召回列表合成稳定排序。
MMR降低结果重复，保留多样性。Agent 上下文窗口有限，重复 facts 会浪费 token。
Cross encoder更贵但更准的语义重排。用于最终候选重排，避免对全库做昂贵 LLM 判断。
Node distance围绕某个实体找近邻事实。个性化或局部上下文检索时比全局相似度更贴近问题。

核心设计思想

Episode First先保存原始输入，fact 永远可追溯。
Temporal Edge事实在边上，边有 valid_at / invalid_at / expired_at。
LLM as Semantic OperatorLLM 做抽取、判重、矛盾识别，系统负责候选、索引、落库。
Hybrid RetrievalBM25、embedding、BFS、reranker 可组合。
Pluggable Infrastructuredriver、LLM、embedder、reranker 都可替换。
Schema 可控entity_types / edge_types 用 Pydantic model 约束业务语义。

真实例子和边界

用户先说：“源码分析先讲仓库边界，再讲主流程。”后面又说：“分享时别先讲边界了，先讲应用场景。”

await graphiti.add_episode(
    name="preference",
    episode_body="用户希望源码分析先讲仓库边界，再讲主流程。",
    source_description="chat message",
    reference_time=now,
    group_id="share",
)
Graphiti 的处理：第一句话形成旧偏好 fact；第二句话形成新偏好 fact；旧 fact 可能被写入 invalid_at，从当前事实变成历史事实。

更真实的业务例子：客户偏好随时间变化

旅行助手里，用户 3 月说：“我喜欢行程排满一点，酒店便宜就行。”6 月又说：“这次带孩子，行程别太赶，酒店要安静。”这不是简单覆盖，因为系统既要知道当前偏好，也要保留历史变化原因。

await graphiti.add_episode(
    name="travel-preference-march",
    episode_body="用户喜欢紧凑行程，酒店优先便宜。",
    reference_time=datetime(2026, 3, 10),
    group_id="travel-user-42",
)

await graphiti.add_episode(
    name="travel-preference-june",
    episode_body="用户这次带孩子旅行，偏好轻松行程和安静酒店。",
    reference_time=datetime(2026, 6, 18),
    group_id="travel-user-42",
)
事实变化Graphiti 表达为什么这么设计
“酒店便宜优先”变成历史偏好。旧 edge 写入 invalid_at 或被新事实替代。回答当前问题时不误用过期偏好。
“安静酒店”成为当前偏好。新 edge 带 valid_at 和 episode 证据。能解释推荐依据来自哪次对话。
“带孩子”是本次旅行上下文。实体和关系连接到同一 group_id。检索时能把人物、偏好、时间和证据一起召回。
为什么适合 Graphiti：Graphiti 的核心不是记一条 memory，而是把 episode、entity、relation 和时间有效性一起建模。它适合“事实会变化、还要解释历史依据”的场景。
局限性说明
依赖 structured output小模型或 schema 支持弱的模型可能导致抽取失败。
写入链路重包含 LLM、embedding、search、dedupe、落库，适合异步队列。
运维成本高需要图数据库、索引、LLM provider、队列和监控。
不是托管 Zep用户/线程/dashboard/SLA/治理需要 Zep 或自行建设。
