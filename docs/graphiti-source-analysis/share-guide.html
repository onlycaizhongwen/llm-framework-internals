Graphiti 源码分享讲稿

面向分享的精读版：先讲仓库边界，再讲 temporal graph memory 主流程，最后和 mem0 / Zep / LangGraph 建立对比。

开场定位

今天讲 Graphiti。它不是 Zep Cloud adapter，也不是 LangGraph workflow runtime，而是 Zep 背后的开源 temporal context graph engine。最重要的关键词是：episode、entity、fact edge、valid_at、invalid_at、hybrid search。
这句话可以展开成三层：有来源，每条事实能追溯到 episode；有时间窗口，事实有 valid_at、invalid_at、expired_at；可混合检索，不是只靠 embedding，而是 BM25、向量、BFS 和 rerank 组合。

一句话主线

add_episode
  -> 保存 EpisodicNode 作为 provenance
  -> LLM 抽 entity 和 fact edge
  -> resolve/dedupe/contradiction
  -> 旧 fact 写 invalid_at / expired_at
      -> search/search_ 用 BM25 + vector + BFS + rerank 检索
为什么这么设计：episode 保留证据，entity 抽出对象，fact edge 表达关系和时间窗口。LLM 做语义判断，工程代码负责候选、索引、分区、落库和可观测性。

源码精读口径

Graphiti 主类：graphiti_core/graphiti.py:137 初始化 driver、LLM、embedder、cross_encoder；:980 是 add_episode()；:1527 是基础 search()；:1584 是高级 search_()。
数据模型：nodes.py:318 是 episode；nodes.py:499 是 entity；edges.py:263 是 fact edge。
时间失效：edge_operations.py:545-573 给旧 fact 写 invalid_at 和 expired_at。
混合检索：search.py:98-244 并行搜多个 scope；search.py:262-462 展示 edge search 的 BM25、cosine、BFS 和 rerank。
去重细节：EntityNode 先语义候选再确定性相似/LLM resolve；EntityEdge 先 exact fact 快速去重，再查 related edges 和 invalidation candidates，因为 fact 既可能重复，也可能推翻旧事实。

对比讲法

项目一句话定位
Graphiti开源 temporal graph memory engine。
Zep托管 context graph infrastructure，Graphiti 是开源核心之一。
mem0轻量 memory layer，更快接入长期记忆。
LangGraphAgent workflow runtime，可和 Graphiti 组合。

局限性

  • 依赖 LLM structured output，小模型或弱 schema 支持会影响抽取。
  • 写入链路重，适合异步队列，不适合阻塞用户请求。
  • 自托管需要图数据库、索引、LLM provider、监控和队列。
  • Graphiti 不等于 Zep 托管平台能力。
  • Kuzu 已 deprecated，新项目优先 Neo4j/FalkorDB。
