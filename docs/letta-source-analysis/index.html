1. 总体结论

Letta 的核心定位是 stateful agent server：它把 Agent 的 persona、人类画像、工具、消息、长期记忆、文件知识、运行状态都放进服务端数据模型，再通过 REST、OpenAI-compatible API 和 streaming 暴露出去。

一句话分享：LangGraph 强在“流程状态图怎么流转”，mem0 强在“记忆事实怎么抽取和检索”，Letta 强在“把一个有身份、有核心记忆、有工具、有长期档案的 Agent 做成服务端对象，并让它可以持续对话和自我维护记忆”。
版本边界：README.md 明确说明该仓库是 legacy Letta server，活跃开发迁到 letta-code。因此这份分析聚焦 V1 API/server、Agent Runtime、memory、tools 和 provider adapter。
主线为什么值得讲
FastAPI router -> SyncServer -> Managers看懂服务端分层，知道 API 为什么不直接操作数据库。
AgentStateLetta 的 Agent 是持久资源，不是一次性 prompt。
Agent.step / inner_step一次消息如何进入 LLM、工具、消息持久化和总结。
Memory 分层core blocks 常驻 prompt，archival/source passages 按需检索。
Tool-first loop回复用户、改记忆、搜档案、调 MCP 都统一成 tool call。

3. AgentState：核心对象

读 Letta 源码时，最先要抓住 AgentState。它说明 Letta 的 Agent 是数据库里的长期资源，有配置、有记忆、有工具、有运行规则。

字段作用为什么重要
llm_config模型、endpoint、context windowProvider adapter 和上下文预算都依赖它。
embedding_config长期记忆和文件 passage 的 embedding 配置决定 archival/source 检索能力。
blocks / memorypersona、human、自定义核心记忆块高频事实常驻 system prompt。
toolsAgent 可调用的工具列表回复、记忆编辑、文件检索都工具化。
tool_rules工具顺序和终止规则降低乱调工具风险。
enable_sleeptime后台记忆维护模式把主对话和记忆整理分离。

设计范式Agent-as-stateful-resource。LangChain 里很多 Agent 是运行时组合；Letta 则把 Agent 做成有 ID、有配置、有记忆、有工具、有运行记录的服务端资源。

7. Provider Adapter

Letta 的 provider 适配是 Factory + Adapter + Provider Registry。Agent runtime 不直接依赖某个 SDK，而是通过 LLMClientBase 统一请求、streaming、usage 和 embeddings。

class LLMClient:
    @staticmethod
    def create(provider_type: ProviderType, actor: User) -> LLMClientBase:
        match provider_type:
            case ProviderType.openai:
                return OpenAIClient(...)
            case ProviderType.anthropic:
                return AnthropicClient(...)

8. Streaming、Run、Step 和 Summarizer

服务端运行状态

RunManager、StepManager、StreamingService 让 Letta 像后端平台：支持 SSE、后台流、取消、keepalive、错误收束和状态更新。

证据：streaming_service.py:196、run_manager.py:48、run_manager.py:320、step_manager.py:40。

上下文自管理

当 usage 接近 context window 或发生 overflow，Agent 会触发 summarizer。它压缩的是 Agent 历史消息，而不是像 Headroom 那样作为外部代理压缩工具输出。

证据：agent.py:945-987、agent.py:1039、summarizer_sliding_window.py:99。

9. 真实例子：源码分析助手

假设我们要做一个长期服务的“源码分析助手”，用户偏好是中文、先架构后主流程、HTML 为主、图中写中文说明和为什么、浅色背景。

agent = AgentCreate(
    name="source-analysis-assistant",
    memory_blocks=[
        {"label": "persona", "value": "我是一个擅长开源 LLM 框架源码精读的中文助手。"},
        {"label": "human", "value": "用户主要看 HTML，偏好浅色背景、中文说明、先架构后主流程。"},
    ],
    include_base_tools=True,
)

当用户说“分析 letta-ai/letta，参考其他工程”时，Letta 会读取 AgentState，把 persona/human blocks 编进 system prompt。模型可以调用文件工具读源码，调用 archival memory 找之前 mem0/headroom 的分析风格，也可以通过 memory 工具更新用户偏好。

这个例子能说明 Letta 和 mem0 的差别：mem0 更像外部 memory layer；Letta 更像把 memory、tools、messages、LLM provider 和 execution lifecycle 都封装进 Agent server。

11. 局限性和阅读注意点

局限 / 注意点为什么建议
当前仓库是 legacy serverREADME 已说明活跃开发迁到 letta-code分享时先讲清版本边界。
历史包袱较重有 deprecated memory 字段、V1 API、旧 MemGPT 概念主线放在 AgentState、Agent loop、Blocks/Passages、ToolExecution。
