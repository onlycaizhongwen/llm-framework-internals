1. 一句话定位

OpenAI Agents Python 是一个轻量但覆盖面很宽的多 Agent SDK。它不是 Dify 这种低代码平台，也不是 LangGraph 这种显式状态图运行时；它更像“代码优先的 Agent loop + OpenAI 原生模型能力 + 工具、handoff、guardrail、session、tracing、realtime、sandbox 的 SDK 套件”。

为什么这么设计：源码把最常用的 Agent 工作流收敛到 Agent 和 Runner，让开发者用 Python 对象声明 instructions、tools、handoffs、guardrails、output_type，再由 Runner 统一处理模型调用、工具执行、转交、流式事件、会话持久化和 tracing。

5. Agent 设计：声明式配置对象

Agent 是 dataclass，核心字段包括 instructions、prompt、tools、mcp_servers、handoffs、model、model_settings、input/output guardrails、output_type 和 tool_use_behavior。

模型配置

model 和 model_settings 让 Agent 选择不同模型实现和采样参数。

工具与 MCP

tools 是本地或 hosted tools，mcp_servers 会在运行时拉取工具列表。

输出契约

output_type 支持 Python type、Pydantic、TypedDict 或自定义 schema。

运行策略

tool_use_behavior 决定工具结果回灌模型还是直接成为最终结果。

源码证据：src/agents/agent.py:173 定义 AgentBase，agent.py:270 定义 Agent；agent.py:285-379 覆盖 instructions、prompt、handoffs、model、guardrails、output_type 和 tool_use_behavior。

10. 真实示例：客服分流 + 工具 + Guardrail

from dataclasses import dataclass
from agents import Agent, Runner, GuardrailFunctionOutput, input_guardrail, function_tool

@dataclass
class UserContext:
    user_id: str
    vip: bool

@input_guardrail
def block_sensitive(ctx, agent, user_input):
    text = user_input if isinstance(user_input, str) else str(user_input)
    blocked = "导出全部用户" in text
    return GuardrailFunctionOutput(
        output_info={"reason": "敏感批量数据导出"} if blocked else {},
        tripwire_triggered=blocked,
    )

@function_tool
def lookup_order(order_id: str) -> str:
    return f"订单 {order_id} 当前状态：已付款，待发货"

billing_agent = Agent(
    name="Billing",
    instructions="处理账单、退款、发票问题。",
    tools=[lookup_order],
)

triage_agent = Agent(
    name="Triage",
    instructions="判断客户问题类型，账单问题转交 Billing。",
    handoffs=[billing_agent],
    input_guardrails=[block_sensitive],
)

result = Runner.run_sync(
    triage_agent,
    "帮我查一下订单 A100，并看看能不能退款",
    context=UserContext(user_id="u1", vip=True),
)
print(result.final_output)

这个例子对应三条源码主线：handoff 被包装成 transfer_to_billing 工具；lookup_order 被包装成 FunctionTool 并进入 ToolExecutionPlan；guardrail 的 tripwire 触发后中断运行。

13. 专题一：Streaming 细节

flowchart TB App["业务代码调用 Runner.run_streamed
为什么：前端和服务端要边生成边响应"] --> Result["RunResultStreaming
保存 current_agent、final_output、interruptions、队列"] Result --> Queue["asyncio event_queue
为什么：run loop 后台推进，消费端异步读取"] Model["模型 stream_response"] --> Raw["RawResponsesStreamEvent
原始 Responses 流事件，适合 token/UI 打字机"] Raw --> Queue Step["run loop 解析模型输出"] --> Items["RunItemStreamEvent
message、tool_called、tool_output、handoff、MCP approval"] Items --> Queue Handoff["发生 handoff"] --> AgentUpdate["AgentUpdatedStreamEvent

raw 负责即时文本，RunItem 负责可解释步骤，AgentUpdated 负责状态栏"]

源码里 Streaming 不是“把 token 原样吐出去”，而是三类事件并行服务不同目标：RawResponsesStreamEvent 做即时文本，RunItemStreamEvent 做工具、handoff、MCP approval 等语义步骤，AgentUpdatedStreamEvent 做多 Agent 切换提示。

为什么这么设计：原始 token 流解决“快”，语义 RunItem 解决“可解释”，AgentUpdated 解决“多 Agent 状态可见”。如果只暴露 token，工具调用、handoff、审批、MCP 列表都会变成黑盒。

  • src/agents/result.py:445-500 定义 RunResultStreaming，包含 _event_queue、guardrail queue、interruptions 和 _last_processed_response。
  • src/agents/stream_events.py:10-61 定义 raw response、run item、agent updated 三类 StreamEvent。
  • src/agents/run_internal/streaming.py:28-65 把 message、tool call、tool output、handoff、MCP approval 映射成语义事件，并跳过 ToolApprovalItem。
  • src/agents/run_internal/run_loop.py:589 运行开始时先发 AgentUpdated；run_loop.py:1484-1485 把模型原始流事件包装成 RawResponsesStreamEvent。
  • tests/fastapi/streaming_app.py:21 演示用 NDJSON 把 stream events 暴露给 HTTP 客户端。

分享口径：OpenAI Agents Python 的 Streaming 是“前台体验 + 后台可观测”一体化设计。前端拿 raw event 做即时输出，工程侧拿 run item event 做工具链路、handoff、MCP 审批可视化。

14. 专题二：Resume / HITL 细节

flowchart TB Call["模型请求工具"] --> NeedApproval{"工具需要审批吗"} NeedApproval -->|否| Execute["直接执行工具
进入 ToolExecutionPlan"] NeedApproval -->|是| Approval["ToolApprovalItem
记录 raw tool call、tool_name、namespace"] Approval --> Interrupt["NextStepInterruption
为什么：暂停 run，但保留可恢复状态"] Interrupt --> Human["人审或策略审批

读取 last_processed_response 和 generated_items"] Resume --> Plan["build_plan_for_resume_turn

Resume / HITL 的核心是：审批不是异常退出，而是运行时的一种合法下一步。模型已经产出了工具调用，SDK 把需要审批的调用放进 ToolApprovalItem，再用 NextStepInterruption 暂停 run。

为什么这么设计：人审场景最怕“重新跑一遍模型导致工具调用漂移”。源码保存上一轮模型响应和生成项，在 resume turn 里重建执行计划，保证恢复时执行的是原先等待审批的工具。

  • src/agents/items.py:509-535 定义 ToolApprovalItem，保存 raw tool call、tool_name、tool_namespace、tool_origin 等审批追踪信息。
  • src/agents/items.py:632-635 明确 ToolApprovalItem 不能直接转成 API input，说明它是 SDK 内部恢复/审批状态。
  • src/agents/run_internal/run_steps.py:170-174 定义 NextStepInterruption，代表工具审批导致的 run 暂停。
  • src/agents/run_internal/tool_planning.py:178-193 的 ToolExecutionPlan 保存 pending interruptions、approved MCP responses、callback approval requests。
  • src/agents/run_internal/tool_planning.py:266-299 的 _build_plan_for_resume_turn 专门为恢复轮次构建执行计划。
  • src/agents/run_internal/run_loop.py:731-800 检测中断后调用 resolve_interrupted_turn，恢复后仍有审批则再次 finalize interruption。

真实例子：客服退款 Agent 调用 refund_order(order_id, amount) 前需要人工确认。主管批准后 resume，同一个 tool call 被执行；如果还有“发票作废”审批，则再次 interruption，而不是丢失上下文。

同时沉淀审计证据和可复盘步骤"]

推荐组合边界：Dify / Agno / 自研前台负责入口、权限、表单、发布和运营；LangGraph 做外层确定性流程和 checkpoint；OpenAI Agents Python 做内层智能体执行；Haystack / LlamaIndex 作为 RAG 工具后端；Tracing 串联每次 Agent run 的工具调用、handoff、审批和模型响应。

为什么这么组合：LangGraph 擅长“流程确定性和恢复”，Dify/Agno 擅长“产品入口和运营”，OpenAI Agents Python 擅长“内层 Agent loop 和 OpenAI 原生能力”。不要让一个框架承担所有职责，边界清楚后，分享和落地都会更稳。

演讲口径：如果听众问“有 LangGraph 了为什么还要 OpenAI Agents Python”，可以回答：LangGraph 管外层流程状态，OpenAI Agents 管节点内部的模型、工具、handoff、guardrail、MCP、streaming、tracing。两者不是替代关系，更像“流程引擎 + Agent 执行单元”。

17. 设计思想总结

声明式 Agent

Agent 只是配置，不直接执行；执行统一交给 Runner。

工具化控制流

handoff、MCP、hosted tool 都进入统一工具集合。

Runtime 承担确定性

模型负责选择，SDK 负责 schema、approval、guardrail、session、trace、resume。

OpenAI 原生优先

Responses、Conversation、WebSocket、tracing、agent registration 被纳入 provider。

可扩展边界清晰

Model、ModelProvider、Session、MCPServer、TracingProcessor 都可替换。

生产调试优先

RunItem 和 Span 让一次 agent run 可以被回放、解释和定位。

分享口径：OpenAI Agents SDK 不是“又一个 prompt wrapper”，它标准化了 Agent 应用最常见的控制点。复杂状态机交给 LangGraph，产品入口交给 Dify/Agno，自身作为内层 agent loop 非常合适。
