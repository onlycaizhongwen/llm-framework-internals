LangChain 源码分享讲解稿

配合主分析页使用，聚焦 Agent middleware、provider adapter、RAG 细节、classic 迁移关系。

开场定位

现在读 LangChain 源码，不能再把它当成一个单体大包。它已经拆成协议层、应用入口层、供应商适配层和 classic 兼容层。当前 Agent 是基于 LangGraph 编译出来的状态图，middleware 是扩展点；RAG 则是 Document、Embedding、VectorStore、Retriever 和 ChatModel 的组合。

讲解顺序

  1. 包边界：langchain-core 定协议，langchain 做当前主入口，partners 接 provider，langchain-classic 保留旧 API。
  2. Agent middleware：create_agent 编译 StateGraph，hook 变节点，wrapper 包调用。
  3. Provider adapter：init_chat_model 解析 provider，partner 包翻译外部 SDK。
  4. RAG：TextSplitter -> Document -> Embeddings -> VectorStore -> Retriever -> Prompt -> ChatModel。
  5. Classic 迁移：旧 chains、memory、classic agents 留在 langchain-classic。

四个专题话术

Agent middleware

create_agent 的本质是图构建器。middleware 分两类：before/after 生命周期 hook 会变成图节点；wrap_model_call/wrap_tool_call 会包住模型或工具调用，用来做重试、限流、缓存、路由和审计。

Provider adapter

主包不直接绑定 OpenAI SDK。init_chat_model 解析 provider:model，langchain-openai 负责把 Message、Tool、stream chunk 翻译成 OpenAI SDK 的请求和响应。

RAG 细节

RAG 不是黑盒，而是一串协议对象：切分器生成 Document，Embedding 生成向量，VectorStore 写入和搜索，Retriever 统一检索入口，最后把召回内容交给 ChatModel。

Classic 迁移

libs/langchain 现在发布为 langchain-classic，是旧 API 的兼容层。新项目优先看 libs/langchain_v1 和 libs/core。

口头总结

LangChain 的源码主线可以压缩成三句话：第一，langchain-core 定义统一协议，所以模型、工具、检索器、向量库都能被组合；第二，当前主包 langchain 把这些协议组织成模型入口和基于 LangGraph 的 Agent；第三，外部 provider 都通过 partner 包适配，classic 包则保留旧 API。
