LangChain 源码架构与专题精读

本页补充四个专题：Agent middleware、provider adapter、RAG 细节、classic 迁移关系。分析对象是本地 sources/langchain，当前提交 eb2dabb8b7102fbedb33016dcf10fe475efde88e，版本线索为 langchain 1.3.4。

主包：libs/langchain_v1 协议：libs/core 适配：libs/partners 兼容：libs/langchain

总体结论

LangChain 当前 Python 仓库不是一个单体包，而是一个多包 monorepo。langchain-core 定协议，langchain 组装 Agent 和模型入口，partners 接外部 provider，langchain-classic 承接旧 API。

一句话概括：现在读 LangChain，要先看协议层和新 Agent 图编排，再看 provider adapter，最后回到 classic 理解旧 API 和迁移背景。

专题一：Agent middleware

新 Agent 的入口在 libs/langchain_v1/langchain/agents/factory.py:create_agent。它不是直接运行一个函数链，而是创建并编译一个 LangGraph StateGraph。

源码证据说明
factory.py:697-718create_agent 接收 model、tools、middleware、response_format、checkpointer、store、interrupt 等参数。
factory.py:823-829模型输出 tool calls 后进入 tools 节点，工具结果以 ToolMessage 回到消息列表，再次调用模型。
factory.py:1047-1055创建 StateGraph。
factory.py:1386-1390添加 model 节点和 tools 节点。
factory.py:1392-1474middleware 的 before/after hook 被编译为独立图节点。
factory.py:1516-1551添加 tools -> model、model -> tools 的条件边。

Middleware 的两类扩展面

类型方法运行方式适合做什么
生命周期节点before_agent、before_model、after_model、after_agent被编译成 LangGraph 节点注入状态、裁剪消息、人工确认、结果整理。
调用包装器wrap_model_call、wrap_tool_call包住 handler，可调用一次、多次或短路重试、限流、缓存、动态路由模型、工具错误处理。
 
class AgentMiddleware(Generic[StateT, ContextT, ResponseT]):
    """Base middleware class for an agent.

    Subclass this and implement any of the defined methods to customize agent behavior
    between steps in the main agent loop.
    """

wrap_model_call 的源码文档在 types.py:491-503 说明：middleware 可以多次调用 handler 做 retry，可以跳过 handler 做 short-circuit，也可以修改 request/response。wrap_tool_call 在 types.py:662-730 同样支持工具请求改写和重试。

设计含义：Agent 主干图稳定，扩展能力用 middleware 注入。

专题二：Provider adapter

init_chat_model 定义在 libs/langchain_v1/langchain/chat_models/base.py:210。它的职责不是调用模型，而是把用户传入的 model 字符串和 provider 参数解析成具体 BaseChatModel 实例。

OpenAI adapter 的核心是 BaseChatOpenAI(BaseChatModel) 和 ChatOpenAI(BaseChatOpenAI)，位置在 libs/partners/openai/langchain_openai/chat_models/base.py。

源码位置适配职责

openai/base.py:1549-1608流式调用 OpenAI SDK，并把 chunk 转成 LangChain chunk。

Provider adapter 的核心价值：主包不直接绑 SDK，上层面向 BaseChatModel，底层 provider 包负责参数、消息、工具、响应、流式和元数据转换。

如果你现在用的是建议迁移方向原因
classic chains改成 LCEL/Runnable 或显式函数组合新主包更强调 Runnable 与图编排。
classic agents迁到 create_agent当前 Agent 基于 LangGraph，支持 middleware、checkpoint、interrupt、store。
old memory迁到 LangGraph checkpointer/store 或外部 memory 系统create_agent 参数里已有 checkpointer 和 store。
旧 provider import使用 partner 包，如 langchain-openaiprovider 适配已拆到独立包。
旧 RAG helper chain拆成 TextSplitter、Embeddings、VectorStore、Retriever、Prompt、Model更透明，也更容易替换每个环节。

真实例子：企业知识库 Agent

一个源码分析知识库助手可以这样串起来：先把 Markdown 分析文档切成 Document，写入向量库；用户提问时由 retriever 召回相关片段；Agent 把 retriever 包成工具；middleware 在模型调用前裁剪历史消息，在工具调用失败时重试；模型通过 init_chat_model("openai:gpt-5.5") 或可配置模型入口创建。

部分对应源码主线
模型可替换init_chat_model + provider adapter
RAG 可替换TextSplitter / Embeddings / VectorStore / Retriever
Agent 可扩展create_agent + AgentMiddleware
旧代码迁移classic chains 拆成新协议组合

分享建议

  1. 先讲包边界：core -> langchain_v1 -> partners -> classic。
  2. 再讲一条主线：create_agent 编译 LangGraph，不是传统 chain。
  3. 然后讲 middleware：生命周期 hook 变节点，wrap hook 包调用。
  4. 再讲 provider adapter：主包不直接绑 SDK，partner 包负责翻译。
  5. 接着讲 RAG：它不是一个大模块，而是多个 core 协议组合。
  6. 最后讲 classic：旧 API 仍在，但新项目不应从 classic 开始。
开场白：现在读 LangChain 源码，最重要的是别把它当成一个大包。它已经分成协议层、应用入口层、供应商适配层和 classic 兼容层。当前 Agent 是基于 LangGraph 编译出来的状态图，middleware 是扩展点；RAG 则是 Document、Embedding、VectorStore、Retriever 和 ChatModel 的组合。
