1. 总体架构

LangChain 当前 Python 仓库是一个多包 monorepo。它不是把所有能力塞进一个包，而是分成“核心协议、当前主包、经典兼容包、供应商集成、文本切分、标准测试”几条分支。

怎么读这张图：从左上角“应用开发者”开始看。应用代码通常进入 libs/langchain_v1，再分成两条主线：create_agent 构建智能体图，init_chat_model 初始化模型。底部的 langchain-core 是所有分支共同依赖的协议层。

2. 源码分支说明

如果你只想分析当前版本主流程，优先看 libs/langchain_v1 和 libs/core。如果你看到旧式 Chain、Memory、Classic Agent，再去 libs/langchain。

corelibs/core

这是“接口层”。LangChain 的模型、工具、检索器、向量库都先在这里定义统一协议，再由不同实现接入。

mainlibs/langchain_v1

这是当前主包。create_agent 在这里把模型、工具、中间件编译成 LangGraph 图。

classiclibs/langchain

这是旧 API 兼容区。它还很重要，但不是当前主流程的第一入口。

partnerslibs/partners

这是供应商适配区。例如 OpenAI 包把 OpenAI SDK 的请求响应转换成 LangChain 的 BaseChatModel、Embeddings 等接口。

splitlibs/text-splitters

这是文本切块工具。RAG 场景里，原始文本通常先在这里切成 Document，再进入向量化和检索。

testslibs/standard-tests

这是集成包测试基座。它保证不同供应商实现同一套 LangChain 行为契约。

3. Agent 主流程

create_agent 不是普通函数链，而是图构建器。它把模型调用、工具调用和 middleware hook 编译成 LangGraph 的节点和边。

流程说明：用户调用 agent 后，状态里带着历史消息和上下文。模型节点先调用 ChatModel；如果模型返回工具调用，流程进入 tools 节点执行工具，再把工具结果作为 ToolMessage 喂回模型。这个循环持续到模型不再请求工具，或结构化输出/中间件把流程导向结束。
flowchart TD
    Start([用户调用已编译 Agent]) --> State["Agent 状态\nmessages + context"]
    State --> BeforeAgent{"是否有 before_agent\n中间件？"}
    BeforeAgent --> BeforeModel{"是否有 before_model\n中间件？"}
    BeforeModel --> ModelNode["模型节点 model\n构造 ModelRequest"]
    ModelNode --> WrapModel["wrap_model_call 调用栈\n重试 / 缓存 / 路由 / 修改请求"]
    WrapModel --> ChatModel["Chat 模型调用\nBaseChatModel.invoke / stream"]
    ChatModel --> AIMessage["模型返回\nAIMessage"]
    AIMessage --> AfterModel{"是否有 after_model\n中间件？"}
    AfterModel --> HasTools{"模型是否发起\ntool calls？"}
    HasTools -- 否 --> Structured{"是否需要\n结构化输出？"}
    Structured -- 否 --> AfterAgent{"是否有 after_agent\n中间件？"}
    Structured -- 是 --> StructuredTool["结构化输出工具"]
    StructuredTool --> AfterAgent
    HasTools -- 是 --> ToolNode["工具节点 tools"]
    ToolNode --> WrapTool["wrap_tool_call 调用栈"]
    WrapTool --> BaseTool["工具执行\nBaseTool.run / invoke"]
    BaseTool --> ToolMessage["工具结果\nToolMessage"]
    ToolMessage --> ModelNode
    AfterAgent --> End([结束并返回响应])

4. RAG / 检索增强流程

RAG 不是单独巨型模块，而是多个 core 协议的组合：Document、TextSplitter、Embeddings、VectorStore、Retriever、Prompt/Model。

流程说明：左侧是建库过程：原始文本先切成 Document chunks，再向量化并写入向量库。右侧是查询过程：用户问题被向量化后做相似度检索，相关文档再和问题一起组装成提示词，交给 ChatModel 生成答案。
flowchart LR
    Raw["原始文本 / 文件"] --> Splitter["文本切分器 TextSplitter\nsplit_text / create_documents"]
    Splitter --> Documents["文档块 Document chunks\npage_content + metadata"]
    Documents --> EmbedDocs["文档向量化\nEmbeddings.embed_documents"]
    EmbedDocs --> Store["写入向量库\nVectorStore.add_texts / add_documents"]
    Store --> Retriever["检索器\nVectorStoreRetriever / BaseRetriever"]
    Query["用户问题"] --> EmbedQuery["问题向量化\nEmbeddings.embed_query"]
    EmbedQuery --> Search["相似度检索\nVectorStore.similarity_search"]
    Retriever --> Search
    Search --> RelevantDocs["相关文档"]
    RelevantDocs --> Prompt["组装提示词 / 消息\nPrompt / messages"]
    Prompt --> ChatModel["生成模型\nBaseChatModel"]
    ChatModel --> Answer["最终回答"]

5. 真实例子：企业知识库问答助手

假设产品要做一个“内部源码知识库问答助手”，用户问：“Graphiti 和 mem0 的区别是什么？帮我用中文说明，最好能引用源码分析文档。”

用户问题
  -> Retriever 检索 docs/mem0-source-analysis 与 docs/graphiti-source-analysis
  -> Prompt 把问题、检索片段、回答格式约束组装成 messages
  -> ChatModel 生成中文答案
  -> Runnable 链路统一支持 invoke / stream / batch
源码抽象在例子里的角色为什么这么设计
Document保存 Markdown/HTML 切片和 metadata。把“内容”和“来源”一起传递，答案才能引用出处。
TextSplitter把长文档切成适合 embedding 的 chunks。避免整篇文档过长，也让检索粒度更细。
Embeddings把 chunk 和 query 转成向量。把自然语言问题变成可做相似度搜索的数据结构。
VectorStoreRetriever按语义找回相关文档片段。让上层链路不关心具体向量库是 FAISS、Chroma 还是云服务。
ChatModel根据上下文生成最终回答。模型调用被统一成标准接口，方便替换 provider。
Runnable把检索、组装 prompt、模型调用串成链。同一套协议支持同步、流式、批处理和组合。
分享讲法：LangChain 的价值不是“替你写一个 RAG 函数”，而是把 RAG 拆成可替换的协议层。真实项目里最常变的是模型、向量库、检索策略和 prompt；这些都被抽象成组件后，业务代码不用跟每个 provider 绑死。

  1. libs/core/langchain_core/runnables/base.py：先理解统一执行协议。

  7. libs/core/langchain_core/retrievers.py 与 vectorstores/base.py：理解 RAG 抽象。
