LangGraph 源码架构分析

先架构，后脉络：把 LangGraph 看成“状态图编译 + Pregel 执行时 + checkpoint 持久化”的低层编排框架。

源码：sources/langgraph 提交：d57a74f950b87bfb9cb51240cc8dccf34b5edfaa 分享优先 HTML

总体结论

LangGraph 不是 LangChain 那种模型、工具、检索和供应商集成组件库，而是一个图执行运行时。用户用 StateGraph 描述节点、边和状态更新规则，compile() 把图编译成可执行对象，底层由 Pregel 按 step 驱动节点执行，并通过 checkpoint 支持恢复、中断、回放和长期状态。

分享时可以这样讲：LangGraph 的核心不是“写一个 Agent 类”，而是“把 Agent 或工作流建模成状态图”。

核心源码证据

1. StateGraph 是建图 API

源码位置：libs/langgraph/langgraph/graph/state.py:130

class StateGraph(Generic[StateT, ContextT, InputT, OutputT]):
    """A graph whose nodes communicate by reading and writing to a shared state.

    The signature of each node is `State -> Partial<State>`.
    """

含义：节点不直接互相调用，而是读写共享 state。复杂流程被拆成节点、边和状态更新。

2. compile 是声明和执行的分界线

源码位置：libs/langgraph/langgraph/graph/state.py:1164

def compile(...) -> CompiledStateGraph[StateT, ContextT, InputT, OutputT]:
    """Compiles the `StateGraph` into a `CompiledStateGraph` object.

    The compiled graph implements the `Runnable` interface and can be invoked,
    streamed, batched, and run asynchronously.
    """

含义：StateGraph 是 builder，compile 后才得到可运行对象。

3. Pregel 是执行运行时

源码位置：libs/langgraph/langgraph/pregel/main.py:449

class Pregel(...):
    """Pregel manages the runtime behavior for LangGraph applications.

    Pregel combines actors and channels into a single application.
    Actors read data from channels and write data to channels.
    """

含义：运行时把节点看成 actor，把状态传播看成 channel 更新。

4. step 循环驱动执行

源码位置：libs/langgraph/langgraph/pregel/main.py:2979

while loop.tick():
    for _ in runner.tick(...):
        yield from _output(...)
    loop.after_tick()

含义：执行像“回合制”。每一轮选择节点、执行节点、统一应用状态更新。

5. Checkpoint 保存版本化状态

源码位置：libs/checkpoint/langgraph/checkpoint/base/__init__.py:92

class Checkpoint(TypedDict):
    """State snapshot at a given point in time."""

    channel_values: dict[str, Any]
    channel_versions: ChannelVersions
    versions_seen: dict[str, ChannelVersions]

含义：checkpoint 保存 channel 值、channel 版本、节点看过的版本，因此可以恢复、回放和中断继续。

6. Prebuilt Agent 只是 StateGraph 模板

源码位置：libs/prebuilt/langgraph/prebuilt/chat_agent_executor.py:862

workflow = StateGraph(...)
workflow.add_node("agent", RunnableCallable(call_model, acall_model))
workflow.add_node("tools", tool_node)
workflow.add_conditional_edges("agent", should_continue, path_map=agent_paths)
workflow.add_edge("tools", entrypoint)
return workflow.compile(...)

含义：create_react_agent 没有特殊运行时，它也是用 StateGraph 拼出 agent/tool 循环。

设计思想

状态图范式用 state、node、edge 表达复杂流程，而不是把逻辑塞进一个大循环。
编译器/运行时分离compile 前声明图，compile 后统一执行、流式、批处理和异步。
Channel 状态传播channel 决定状态 key 怎么接收、合并和传播更新。
Pregel/BSP step按回合推进，避免并发节点看到半成品状态。
Checkpoint 持久化状态快照和版本关系支撑恢复、回放和人类介入。
Prebuilt 组合Agent 是图模板，核心边界仍是 StateGraph + Pregel。

真实例子：源码分析工作流

把你前面做的源码分析流程抽象成 LangGraph：输入一个 GitHub 仓库地址，系统要自动拉取代码、扫描目录、识别入口、生成架构图、写 Markdown/HTML，最后等待人工复核。

class AnalysisState(TypedDict):
    repo_url: str
    local_path: str
    architecture: dict
    main_flow: list[str]
    docs: dict
    review_status: str

graph.add_node("clone_repo", clone_repo)
graph.add_node("scan_architecture", scan_architecture)
graph.add_node("trace_main_flow", trace_main_flow)
graph.add_node("write_docs", write_docs)
graph.add_node("human_review", human_review)
graph.add_edge("clone_repo", "scan_architecture")
graph.add_edge("scan_architecture", "trace_main_flow")
graph.add_edge("trace_main_flow", "write_docs")
graph.add_edge("write_docs", "human_review")
LangGraph 概念例子里的含义为什么这么设计
State保存 repo、架构、流程、文档、复核状态。所有节点围绕同一份任务状态读写，方便恢复和审计。
Node每一步是独立函数：拉代码、扫目录、写文档。把复杂任务拆成可测试、可重试的小单元。
Edge定义源码分析的推进顺序。流程结构显式化，分享时可以直接画图讲清楚。
Checkpoint写完 HTML 后暂停等待人工看。长任务中断后能从上次状态继续，不必重跑全部步骤。
Conditional edge复核不通过则回到写文档或补扫描。把“人工反馈后返工”的业务分支变成可控流程。
分享讲法：LangGraph 适合讲“任务状态怎么一步步演化”。它不是让 Agent 自由聊天，而是把 Agent 能力放进节点里，再用图、状态和 checkpoint 控制整个长流程。

分享收束

LangGraph 的源码主线其实很清楚：上层用 StateGraph 声明状态图，中间 compile 成可运行图，底层 Pregel 按 step 调度节点，checkpoint 把每一步状态固化下来。Agent 只是这个运行框架上的一种图。
