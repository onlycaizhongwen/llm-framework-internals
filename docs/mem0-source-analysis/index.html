mem0 源码架构精读

参考 AutoGen 的精读方式，但 mem0 的主线不是 Agent 协作运行时，而是长期记忆层：事实抽取、向量写入、历史记录、实体链接、混合检索和可选 rerank。

源码：sources/mem0 提交：cd79fa8914b5b1cf66daacc957d826065df57df8 定位：Memory Layer for AI Agents

总体结论

mem0 是面向 AI 应用和 Agent 的长期记忆框架。它把“记住什么、如何存、如何找回”拆成 Memory Core、LLM/Embedding/VectorStore/Reranker 抽象、SQLite history、entity store 和产品入口。

一句话：AutoGen 解决“多个 Agent 怎么协作”，mem0 解决“Agent 和应用应该记住什么、怎么检索回来”。

校验 scope标准化 messagesLLM 事实抽取embed_batchhash 去重vector inserthistory/entity

Memory.add 是写入主路径。它不是简单把聊天文本 embed 后存起来，而是先让 LLM 抽取长期有价值的 facts，再批量向量化、去重、写入、记录历史和建立实体链接。

读图说明：这张图展示的是“从对话到长期记忆”的写入路径。输入不是直接进入向量库，而是先被标准化为 messages，再结合最近对话和已有记忆，让 LLM 做一次事实抽取，最后才进入 embedding、去重、写库、历史记录和实体链接。
  • 为什么这么设计：聊天原文通常有寒暄、重复、临时上下文，直接存会污染长期记忆；先抽取 facts 可以让 memory 更稳定、更可检索。
  • 为什么要查 existing memories：抽取 prompt 会带已有记忆，目的是减少重复事实，并让 LLM 知道哪些内容已经沉淀过。
  • 为什么要 hash 去重：LLM 仍可能抽出重复 facts，hash 是低成本的确定性防线。
  • 为什么要 entity linking：长期记忆查询常围绕人、地点、项目、偏好展开，实体链接能在后续检索时给相关 memory 加分。

Mermaid 源文件：add-flow.mmd

设计含义：mem0 的 memory 不是普通聊天日志，而是经过 LLM 提炼后的事实记录。它用 hash 去重、history 审计、entity linking 提升长期记忆质量。

源码精读三：Memory.search 混合检索

Memory.search 读取主路径会融合 semantic search、keyword BM25、entity boost 和可选 reranker。它服务的是长期记忆检索，所以要兼顾语义相似、关键词精确和实体一致。

证据：memory/main.py:1331 定义 search()；:1456-1465 调 _search_vector_store()；:1586-1612 做 query 预处理、semantic 和 keyword；:1622-1624 计算 entity boost；:1638-1645 调 score_and_rank()；utils/scoring.py:60 定义融合评分。

读图说明：这张图展示的是“从问题到相关记忆”的检索路径。mem0 先做 query 清洗和作用域校验，再并行取得三类信号：语义向量召回、BM25 关键词匹配、实体链接增强，然后交给 score_and_rank 做融合排序。
  • 为什么不是只做向量检索：长期记忆里常有姓名、产品、地点、具体偏好，仅靠语义可能把精确实体打散；BM25 和 entity boost 用来补足精确匹配。
  • 为什么 threshold 先过滤 semantic：源码里先用 semantic threshold 过滤候选，再叠加 BM25/entity，避免关键词把语义完全不相关的内容强行推上来。
  • 为什么 reranker 是可选：rerank 成本更高，适合对排序质量要求高的场景；普通场景用融合分数已经能覆盖主要需求。

源码精读四：Entity linking

mem0 维护第二个向量集合作为 entity store。写入时抽取实体并记录 linked_memory_ids；检索时抽取 query entities，搜索 entity store，再把相关 memory 加权推高。

证据：utils/entity_extraction.py:751 定义 extract_entities()；:761 定义 extract_entities_batch()；memory/main.py:562 定义 _upsert_entity()；:1685 定义 _compute_entity_boosts()；utils/scoring.py:57 定义 ENTITY_BOOST_WEIGHT = 0.5。

设计含义：实体不是替代向量检索，而是排序增强信号。这样 mem0 能把“关于同一个人、地点、项目或主题”的记忆更稳定地召回。

核心设计思想和范式

Memory Layer 优先核心是记忆生命周期，不是 Agent 编排。
Pipeline 架构_add_to_vector_store 和 _search_vector_store 都拆成稳定阶段。
Ports and AdaptersLLMBase、EmbeddingBase、VectorStoreBase 定义端口。
Factory Method通过 provider name 动态创建 LLM、Embedder、VectorStore、Reranker。
Hybrid Retrievalsemantic、BM25、entity boost 多信号融合。
Session-scoped Memory按 user/agent/run 隔离，降低记忆串线风险。

分享收束
