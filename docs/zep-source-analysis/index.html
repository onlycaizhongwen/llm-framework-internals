Zep 源码架构精读

分析对象：sources/zep，固定源码提交 826c5492d9cc3a7caf92a9870529f29b5a8546e3。这份分析参考 mem0，但会先澄清：当前仓库主要是 Zep Cloud 的 examples、integrations、MCP server、eval harness，不是 Zep Cloud 服务端核心源码。

总体结论

Zep 和 mem0 都在解决 Agent memory，但源码形态不同。mem0 仓库里有本地 Memory core，可以精读 Memory.add/search；当前 getzep/zep 仓库主要是 Zep Cloud 接入层，重点是不同 Agent 框架如何调用 thread.get_user_context、thread.add_messages、graph.add/search。

分享口径：mem0 更像本地 memory layer；当前 Zep 仓库更像 Zep Cloud memory service 的生态接入层集合。

LangGraph 集成精读

LangGraph Python 是最完整的接入包，覆盖 Context Block、message persistence、ZepStore 和 graph search tool。

context.pyget_zep_context() / build_system_message()：取回 Context Block 并组装 system message。
persistence.pypersist_messages()：把本轮消息写入 Zep thread。
store.pyZepStore(BaseStore)：backing store 管 KV，Zep graph 管 semantic search。
tools.pycreate_graph_search_tool()：生成 LangGraph/LangChain 可用工具。
为什么 ZepStore 要 hybrid-delegate：LangGraph Store 有 exact get/put/delete 语义，但 Zep 是 temporal knowledge graph，不是强一致 KV。源码用 backing store 保证 KV 行为，同时把 search 路由到 Zep graph。

证据：store.py:86 定义 ZepStore；:147 分流 op；:159 序列化 put payload；:191 转成 SearchItem；:323 / :348 实现 batch/abatch。

真实例子和边界

用户说：“以后源码分析请用中文，先讲仓库边界，再讲主流程。我正在比较 mem0、Zep 和 LangGraph。”

新一轮对话开始:
  thread.get_user_context(thread_id)
  -> 取回用户长期偏好、项目背景、相关 facts
  -> 注入 system message

本轮对话结束:
  thread.add_messages(thread_id, user + assistant)
  -> Zep Cloud 异步更新 user graph
  -> 下轮 Context Block 可引用这些 facts
和 mem0 的差异：mem0 示例重点是本地 facts 抽取、embedding、去重和 hybrid scoring；Zep 当前仓库示例重点是 framework adapter 如何调用云端 Context Block、graph search 和 thread persistence。

更真实的接入例子：客服 Agent 记住客户偏好

一个电商客服 Agent 中，客户说：“我想给爸爸买鞋，他脚宽，最好防滑，颜色别太花。”本轮回答前，接入层先取 Context Block；本轮结束后，再把 user/assistant turn 写回 thread。

context = zep_client.thread.get_user_context(thread_id=thread_id)
messages = [
    {"role": "system", "content": context.context},
    {"role": "user", "content": "帮我挑一双适合爸爸走路的鞋"},
]
answer = model.generate(messages)
zep_client.thread.add_messages(thread_id=thread_id, messages=[user_msg, answer_msg])
客户事实Zep 接入点产品价值
爸爸脚宽。thread.add_messages 后进入云端图谱处理。下次推荐鞋时自动优先宽楦。
偏好防滑。thread.get_user_context 注入系统上下文。回答不需要用户每次重复说明。
颜色不要太花。graph.search 可作为工具按事实检索。客服 Agent 能解释推荐理由。
为什么适合 Zep：应用侧不用自己维护图谱写入、事实更新和 Context Block 组装，只要在框架扩展点接入读取、写回和搜索。源码分析时要讲清楚：本仓库展示的是这些接入点，不是 Zep Cloud 服务端算法本体。
边界讲法
不是服务端核心源码不能从这里证明云端图谱算法细节。
Context Block 组装在云端本仓库能看到调用点，看不到服务端组装算法。
异步 ingestion不要假设同一 turn 写后立刻能查。
API 限制源码有 truncate、chunk、limit clamp 等工程防线。
隐私治理长期 memory 需要 user/thread 隔离、删除、脱敏和审计。

分享收束

Zep 当前仓库最值得看的不是一个本地 memory class，而是它如何把云端图谱记忆能力包装成各框架可理解的扩展点：LangGraph 的 Store 和 SystemMessage，AutoGen 的 Memory，CrewAI 的 Storage/Tool，Vercel AI SDK 的 middleware/onFinish，以及 MCP tools。
