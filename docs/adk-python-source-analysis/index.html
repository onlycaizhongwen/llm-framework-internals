1. 一句话定位

Google ADK Python 是一个代码优先的 Agent 应用开发框架。ADK 2.0 的特点不是只封装一个 LLM Agent，而是把 Agent、Workflow、Runner、Event、Session、Tool、Task API、Evaluation、ADK Web/API Server/Deploy 放进同一套开发体验里。

为什么这么设计：简单任务用 LlmAgent，复杂确定性流程用 Workflow，需要子 Agent 委派时用 Task API，最后统一落到 Event / Session / Evaluation / API Server。这样既能写代码，又能用 ADK Web 调试和用 eval 做质量回归。

5. Agent 设计：声明式配置 + 可作为 Workflow Node

BaseAgent 继承 Workflow 的 BaseNode。这意味着 Agent 不只是聊天对象，也可以作为 Workflow 图里的节点执行。

  • agents/base_agent.py:93-147 定义 BaseAgent(BaseNode)，包含 name、description、parent_agent、sub_agents。
  • base_agent.py:283-313 的 run_async 负责 before callback、核心实现、after callback。
  • llm_agent.py:223-244 定义默认模型、live 模型和模型继承规则。
  • llm_agent.py:529-570 运行 LLM flow，并在可恢复模式下写 agent end state。
  • llm_agent.py:587-607 把 LlmAgent 包装成 Workflow graph node 执行。

为什么这么设计：ADK 不把“Agent”和“Workflow Node”分成两套世界。简单时独立 chat，复杂时放入流程图，演进成本低。

6. LLM Flow：一次模型调用如何推进

ADK 的 LLM Flow 是“预处理 -> 模型调用 -> 后处理 -> 工具执行 -> 再循环”。

  • flows/llm_flows/base_llm_flow.py:916-929 循环执行 one step，直到 final response。
  • base_llm_flow.py:947-972 在可恢复场景检查 long-running tool call 是否未解决，未解决则暂停。
  • base_llm_flow.py:974-988 恢复后如果最后事件是 function call，则直接处理工具，不重新调模型。
  • base_llm_flow.py:990-1016 调模型并把 LlmResponse 转成 Event。
  • flows/llm_flows/functions.py:546-664 执行单个 function call，包含 callbacks、telemetry、FunctionResponse event。

13. 真实示例：电商售后工单

from google.adk import Agent, Workflow
from google.adk.events.request_input import RequestInput
from google.adk.workflow import node

@node(rerun_on_resume=True)
async def approve_refund(ctx, order_id: str):
    if "refund_approved" not in ctx.state:
        yield RequestInput(
            interrupt_id=f"refund:{order_id}",
            message=f"订单 {order_id} 申请退款，需要主管确认。",
            response_schema=dict,
        )
        return
    return {"approved": ctx.state["refund_approved"]}

triage_agent = Agent(
    name="triage_agent",
    model="gemini-2.5-flash",
    instruction="判断用户问题类型，订单退款交给流程节点处理。",
)

root_agent = Workflow(
    name="ecommerce_after_sales",
    edges=[("START", triage_agent, approve_refund)],
)

这个例子对应四条源码主线：Workflow 编译成 Graph；LlmAgent 作为 Workflow Node 执行；RequestInput 转成 adk_request_input function call；恢复时 rehydration 从 session events 中找到已完成节点和未决 interrupt。

15. 设计思想总结

  1. Agent 和 Workflow 统一到 Node：BaseAgent(BaseNode) 让 Agent 能独立运行，也能成为流程节点。
  2. Event 是系统事实：对话、工具、状态、路由、节点输出、中断都落成 Event。
  3. 恢复依赖事件重放：Workflow 通过 rehydration 扫 session events 重建节点状态。
  4. 子 Agent 工具化：single_turn / task 子 Agent 通过 AgentTool 接入模型 function call。
  5. Google 生态优先：Gemini、Vertex AI、Google API、Agent Engine、Cloud Run、GKE、ADK Web/API Server 是天然主场。
  6. 质量工程内置：Evaluation 和 AgentEvaluator、EvalCase、metrics、pytest/CLI/Web 打通。
  7. 服务后端可替换：Session、Artifact、Memory、Credential 都通过服务层注入，便于从本地调试切到生产部署。
  8. 模型适配解耦：BaseLlm 和 LLMRegistry 让 Agent 运行时不直接绑定某个 provider SDK。

分享口径：ADK 不是“Gemini Agent 的简单封装”，而是 Google 生态下 Agent + Workflow + Web 调试 + Eval + 部署的一体化开发框架。
