TaskWeaver 源码分析

从源码看 TaskWeaver：它是一个 code-first 数据分析 Agent 框架，把自然语言任务转成计划、Python 代码、执行反馈、artifact 和角色协作闭环。

1. 一句话定位

TaskWeaver 是微软开源的 code-first Agent 框架，主场景是数据分析、表格处理、可视化、算法插件编排。它不只保存聊天历史，还把 LLM 生成的 Python 代码、执行结果、内存对象、artifact、插件调用和计划状态都纳入会话。

推荐分享口径：TaskWeaver 的核心不是“让模型直接调用工具”，而是让模型写 Python，并在有状态执行环境里运行 Python。

4. CodeInterpreter：为什么是 code-first

TaskWeaver 的 CodeInterpreter 分四步：CodeGenerator 组装 prompt，LLM 生成结构化 JSON，代码验证检查风险，再由 CodeExecutor 执行 Python 并回填结果。

为什么这么设计：数据分析任务经常依赖 DataFrame、numpy array、模型对象、图表对象。Python 代码比单个 tool call 更适合表达清洗、聚合、建模和可视化组合。

  • taskweaver/code_interpreter/code_interpreter/code_interpreter.py:94：定义 CodeInterpreter。
  • taskweaver/code_interpreter/code_interpreter/code_interpreter.py:113-126：local 模式会自动开启代码验证。
  • taskweaver/code_interpreter/code_interpreter/code_interpreter.py:209-245：验证失败时写入错误，并通过 revise_message 自我修正。
  • taskweaver/code_interpreter/code_interpreter/code_interpreter.py:269：调用 executor.execute_code。
  • taskweaver/code_interpreter/code_interpreter/code_interpreter.py:310-313：执行失败且可重试时回到 CodeInterpreter。

Planner 在 TaskWeaver 里不是一个简单“规划提示词”，它更像任务路由器。它需要输出结构化 JSON，并且 JSON schema 会约束 send_to 只能是 worker alias 或 User。

taskweaver/planner/planner.py:77-80 会把 response_json_schema 的 send_to 枚举收窄到可用 worker 和 User。planner.py:260 内部的 check_post_validity 会检查 send_to、message，并要求 attachment 中必须有 init_plan、plan、current_plan_step。

为什么这么设计：TaskWeaver 的 Planner 不只是 ReAct 里的 thought，而是通过 schema、attachment 和 send_to 共同约束出来的调度层。

这解释了 TaskWeaver 的闭环：失败信息不是给用户看的终点，而是下一次代码生成的输入。

9. 专题三：代码验证与安全边界

TaskWeaver 的安全边界主要由两层组成：代码执行隔离和代码验证。默认执行服务是 container 模式；如果切到 local 模式，CodeInterpreter 会自动打开 verification，因为本机执行风险更高。

taskweaver/code_interpreter/code_verification.py:217 的 separate_magics_and_code 会拆出 magic、shell command 和 pip/conda install；非安装类 magic / shell command 会被拒绝。code_verification.py:253 的 code_snippet_verification 再用 ast.parse 和 FunctionCallValidator 检查 import、函数调用、变量赋值和危险属性访问。

边界要讲清楚：TaskWeaver 更像“代码检查 + 容器隔离”，不是 AgentScope 那种工具级权限、人审、workspace policy。

模型生成的代码可以很直接：

插件封装专业能力，CodeGenerator 负责编排调用顺序，CodeExecutor 负责把插件加载到执行环境，artifact 再回到 Post / Memory 中。

11. 真实例子：销售数据异常分析

  1. 用户上传销售 CSV，提问“最近 30 天销售额为什么下滑？帮我找异常并画图”。
  2. Session.send_message 把文件放入 session 的 execution_cwd。
  3. Planner 生成计划：读 CSV、做缺失值/异常检查、按日期聚合、画趋势图、解释原因。
  4. CodeGenerator 生成 Python：pandas.read_csv、groupby、rolling、matplotlib/seaborn 绘图。
  5. CodeInterpreter 验证代码，通过后交给 Jupyter / container 执行。
  6. 执行结果、图表 artifact、错误日志进入 Post Attachment，Planner 再汇总给用户。

这个例子解释了 TaskWeaver 的核心价值：它适合“需要实际算一遍”的任务，而不是只靠语言模型猜答案。

13. 核心设计思想和范式

Code-first Agent

把 Python 代码作为主要行动语言，适合数据清洗、建模和可视化。

Role-based Collaboration

Planner、CodeInterpreter、ext_role 都是 role，可通过 role YAML 扩展。

Structured Post / Attachment

复杂输出不混在文本里，而是 plan、code、execution_result 等结构化附件。

Reflective Execution

格式、验证、执行失败都能通过 revise_message 回到 CodeInterpreter 修正。

Registry / Adapter

LLM、Role、Plugin、ExecutionService 都通过适配或注册表解耦。

Event-driven UI

状态、消息、附件变化以事件形式推给 CLI/Web UI。

14. 局限性

  • 更适合数据分析和代码执行任务，不是通用企业 Agent 服务框架。
  • 默认 container 执行比纯 SDK 更重，需要 Docker、镜像、依赖和运行环境管理。
  • 权限治理不像 AgentScope 那样细化到人审、工具策略和 workspace 操作模式。
  • 复杂确定性流程不如 LangGraph 清晰。
  • 业务人员低代码配置、发布、运营不如 Dify / Agno 完整。

16. 分享口径

TaskWeaver 是一个典型的 code-first Agent。它不是让模型直接回答数据问题，而是让模型先规划，再写 Python，再在有状态执行环境里运行代码，最后把代码、结果、图表和错误反馈纳入下一步推理。

三条主线

  • 架构主线：App 装配 Session，Session 调度 Planner 和 CodeInterpreter，Memory 用 Round/Post/Attachment 串起角色协作。
  • 执行主线：CodeGenerator 生成代码，CodeInterpreter 验证和重试，CodeExecutor 在 Jupyter / Docker 中执行并回收 artifact。
  • 扩展主线：Plugin YAML 变成 prompt 里的函数签名，Python 实现被注入执行环境，EventEmitter 把状态变化推给 UI。

结尾可以强调：TaskWeaver 适合做数据分析 worker，不一定适合替代 LangGraph、AgentScope、Dify 这类流程、生产、平台框架。
