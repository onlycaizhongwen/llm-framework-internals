AutoGen 源码架构精读

这版不只讲“有哪些模块”，而是沿着 AutoGen 的真实源码路径拆：Runtime 如何投递消息，AssistantAgent 如何做模型和工具循环，GroupChat 如何映射到底层 Core Runtime。

源码：sources/autogen 提交：027ecf0a379bcc1d09956d46d12d44a3ad9cee14 当前状态：maintenance mode

总体结论

AutoGen 的核心不是单个聊天机器人封装，而是一个 消息驱动的多 Agent 运行时。底层 Core 定义 AgentRuntime、消息投递、topic、handler、模型和工具抽象；上层 AgentChat 把这些能力包装成 AssistantAgent、Team 和 GroupChat；Extensions 再接入 OpenAI、MCP、代码执行等生态。

分享时可以这样定调：AutoGen 更像“多 Agent actor runtime + Chat Team API”，而不是单纯的 prompt 编排库。

源码精读一：Runtime 消息投递

send_message / publish_messageEnvelopeasyncio queue_process_next_process_send / _process_publishRoutedAgent handler

Core Runtime 把 Agent 当成消息处理单元。send_message 是点对点请求响应，publish_message 是 topic 广播；二者都会进入 SingleThreadedAgentRuntime 的队列，再由 _process_next 分派到具体处理函数。

证据：_agent_runtime.py:21-50 定义协议；_single_threaded_agent_runtime.py:332 实现 send；:387 实现 publish；:466、:557、:671 处理队列消息。

源码精读二：RoutedAgent 处理器范式

RoutedAgent 通过装饰器把消息类型映射到 handler。@event 表达“收到事件后处理，不要求响应”，@rpc 表达“请求响应”，@message_handler 是更通用的声明式入口。

证据：_routed_agent.py:85 定义 message_handler；:205 定义 event；:325 定义 rpc；:415 定义 RoutedAgent。

class RoutedAgent(BaseAgent):
    """...route messages to handlers based on the type of the message."""

设计含义：这是一种“声明式路由 + 类型驱动 handler”的范式。调用方不用知道具体函数名，只要消息类型和订阅关系正确，runtime 就能把消息送到对应 Agent。

源码精读五：Model / Tool / Workbench 抽象边界

AutoGen 把外部能力压到 Core 协议和 Ext 实现里：Core 定义 ChatCompletionClient、BaseTool、Workbench，Ext 再实现 OpenAI、MCP 等具体适配。这样 AgentChat 不直接绑定 provider。

证据：models/_model_client.py:209-242 定义 ChatCompletionClient.create/create_stream；tools/_base.py:96-179 定义 BaseTool.schema/run/run_json；tools/_workbench.py:78 定义 Workbench；autogen_ext/models/openai/_openai_client.py:1179 实现 OpenAI client；autogen_ext/tools/mcp/_workbench.py:47 实现 MCP Workbench。

class ChatCompletionClient(ComponentBase[BaseModel], ABC):
    async def create(...)
    def create_stream(...)

class BaseTool(ABC, Tool, Generic[ArgsT, ReturnT], ComponentBase[BaseModel]):
    async def run(...)

设计含义：这是典型的 Ports and Adapters。Core 是端口，Ext 是适配器，AgentChat 依赖抽象而不是依赖 OpenAI/MCP 具体实现。

核心设计思想和范式

运行时优先先设计消息、topic、queue、handler，再包装聊天 Agent。证据集中在 AgentRuntime 和 SingleThreadedAgentRuntime。
Actor / Event 模型Agent 是消息处理单元，send_message 和 publish_message 分别对应 RPC 和事件广播。
声明式消息路由@message_handler、@event、@rpc 把消息类型和处理函数绑定，降低调用方耦合。
Template MethodBaseChatAgent.run_stream 固定外部运行协议，子类实现 on_messages_stream。
StrategyRoundRobin、Selector、Swarm 都是在替换 speaker selection 策略。
Ports and Adapters模型、工具、MCP、代码执行通过抽象接口接入，Ext 负责具体 provider。

应用场景和 LangGraph 对比

AutoGen 更适合的场景

多 Agent 研究协作把检索、阅读、总结、审查拆成多个 AssistantAgent，用 RoundRobin 或 SelectorGroupChat 组织对话。
团队式聊天原型需要快速演示“多个专家一起讨论问题”时，Team API 比直接写底层图更快。
人机协同流程UserProxyAgent、终止条件、流式事件适合做 human-in-the-loop 的聊天型应用。
工具和代码执行任务AssistantAgent + Workbench + Code Executor 适合网页浏览、代码执行、文件处理等工具密集任务。
可视化原型和演示Studio 适合先把 Agent、Team、模型配置跑起来，再沉淀成代码。
存量系统维护项目进入 maintenance mode 后，新项目要谨慎，但存量 AutoGen 系统仍需要读懂 Runtime、Team 和 Ext 边界。
选型口径：如果目标是“多个聊天 Agent 通过消息协作”，AutoGen 表达直接；如果目标是“可恢复、可审计、可精确控制的状态流”，LangGraph 更合适。
维度AutoGenLangGraph判断
核心模型AgentRuntime + Message + Team，偏 actor/message runtime。StateGraph + Node + Edge + Checkpoint，偏状态机运行时。对话协作优先 AutoGen；状态流控制优先 LangGraph。
多 Agent 表达RoundRobin、Selector、Swarm 等 Team 模式内建。多 Agent 通常建模为节点、子图、条件边和共享状态。想快速组织“谁发言”用 AutoGen；想明确“状态如何变”用 LangGraph。
流程可控性通过 manager、termination、topic 控制，语义偏群聊。通过显式边、条件路由、interrupt、checkpoint 控制。复杂分支、暂停恢复、审计回放更偏 LangGraph。
状态管理Agent 自身有状态，Team/Manager 可保存状态，但全局状态不是第一入口。全局状态是第一等概念，节点读写状态，checkpoint 可持久化。业务状态强约束时选 LangGraph。
项目建议适合学习 runtime 设计、原型、存量维护。更适合新项目里的复杂 Agent Runtime。新项目优先评估 LangGraph 或 Microsoft Agent Framework。

分享收束

AutoGen 源码最值得看的不是某个 prompt，而是它如何把多 Agent 系统拆成 Runtime、Agent、Message、Team、Model、Tool 几层。底层消息运行时提供通用能力，上层 AgentChat 提供好用入口，Extensions 把外部模型和工具接进来。
